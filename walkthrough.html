UEFI & Bootloader Architecture Fix

Fixed UEFI bootloader generation and boot reliability on bare metal and VM hypervisors.

Fix 1: Host ISO Build Dependency Correction

Added mtools and dosfstools packages to the build containers. This resolved a silent bug where mkarchiso skipped building the UEFI boot image efiboot.img because it lacked FAT/EFI file construction utilities.

Fix 2: Double grub-install Fallback Configuration

Modified the installer script (Step 8) to deploy GRUB twice. The first registers it in standard NVRAM, and the second sets up a removable fallback in the standard EFI directory:

# Installs standard UEFI NVRAM entry
grub-install --target=x86_64-efi --efi-directory=/boot/efi --bootloader-id=CONJUNCTION

# Installs removable/fallback path (essential for virtualized boot and stubborn BIOSes)
grub-install --target=x86_64-efi --efi-directory=/boot/efi --bootloader-id=CONJUNCTION --removable

Unified 'application' Package Manager CLI

Created a user-friendly CLI management tool located at /usr/local/bin/application to unify package discovery and container creation.

Features Supported:

  • application install <name>: Searches Flatpak, Pacman, and yay/AUR sequentially and installs the package, automatically wrapping it in a macOS-style container in /Applications.
  • application uninstall <name>: Detects backend (Flatpak, pacman, or Wine), completely uninstalls system packages, and destroys the corresponding container and launcher.
  • application search <query>: Returns a unified result table aggregating packages across all 3 channels.
  • application list: Lists all installed Flatpaks and user-installed pacman/AUR packages.
  • application info <name>: Returns version, size, source and description details.

Premium macOS .app Containers

Designed a unified application deployment system under the root directory /Applications/ mimicking macOS bundles.

Self-Contained Container Structure

Applications installed via pacman, flatpak, or Wine are structured inside /Applications/<AppName>.app/:

/Applications/Firefox.app/
├── Contents/
│   ├── Info.plist          # Metadata property list
│   ├── MacOS/
│   │   └── Firefox         # Shell launcher script (e.g. exec flatpak run com.discordapp...)
│   └── Resources/
│       ├── icon.png        # Extracted/copied system icon
│       └── prefix/         # (Wine-only) Self-contained Wine prefix
Double-Click Companion Launchers

Since Linux file managers treat directories normally, the installer creates a companion /Applications/<AppName>.desktop launcher next to each bundle folder. This enables instant double-click execution in Dolphin.

Bidirectional Sync Daemon (Rust)

Implemented a highly efficient background sync daemon in Rust to match system packages and the /Applications directory state.

Synchronization Logic:

  • Folder Deleted: If you drag a `.app` container or a `.desktop` companion shortcut to the Trash in /Applications, the daemon immediately uninstalls the corresponding pacman or Flatpak package and cleans up leftover shortcuts.
  • Package Uninstalled: If you run a manual pacman -R or flatpak uninstall via command line, the daemon automatically cleans up the `/Applications` folder, purging the `.app` bundle, companion, and metadata.
